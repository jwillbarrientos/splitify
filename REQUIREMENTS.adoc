== 1. Introducción

=== 1.1 Propósito

Este documento describe los requerimientos funcionales y no funcionales del sistema *Splitify*, una aplicación web que permite a los usuarios categorizar y organizar sus canciones de Spotify por idioma, género musical y artista.

=== 1.2 Alcance

Splitify es una aplicación web que se integra con Spotify y utiliza IA en la categorizacion de musicas. Permite a los usuarios seleccionar playlists existentes (incluyendo Liked Songs) y generar nuevas playlists organizadas según criterios de idioma, género y/o artista.

=== 1.3 Definiciones y Acrónimos

|===
|Término |Definición 

|Liked Songs |Playlist especial de Spotify con las canciones marcadas como favoritas por el usuario
|ChatGPT API |API de OpenAI utilizada para clasificar el género musical y el idioma de cada canción. El modelo usado es *GPT-4.1-mini*
|Tags de Last.fm |Etiquetas curadas por la comunidad de Last.fm. Se obtienen primero a nivel *canción específica* (endpoint `track.gettoptags`, tags como "spanish", "paraguayo", "ballad", "acoustic") y solo como *fallback* a nivel artista (`artist.gettoptags`). Se pasan como contexto a ChatGPT para mejorar la precisión de su clasificación. Last.fm reemplazó a Spotify `/v1/artists` (removido en feb 2026)
|SRS |Software Requirements Specification
|OAuth |Protocolo de autorización utilizado para la autenticación con Spotify
|===

== 2. Descripción General

=== 2.1 Perspectiva del Producto

Splitify actúa como intermediario entre el usuario y Spotify. No almacena música; únicamente lee las playlists del usuario, clasifica las canciones y crea nuevas playlists en la cuenta de Spotify del usuario.

=== 2.2 Usuarios Objetivo

Usuarios de Spotify que desean organizar automáticamente sus bibliotecas musicales según idioma, género o artista sin hacerlo manualmente.

=== 2.3 Restricciones

* Requiere una cuenta activa de Spotify.
* La clasificación de género depende de la disponibilidad de la API de ChatGPT.
* La aplicación se despliega en un servidor Linode (Ubuntu 24.04 LTS) con Caddy como reverse proxy y HTTPS automático via Let's Encrypt.
* El dominio de producción es `splitifyapp.app`.

== 3. Catálogos del Sistema

=== 3.1 Idiomas

Los idiomas no están predefinidos. El sistema detecta dinámicamente el idioma de cada canción mediante consultas a la API de ChatGPT. Para reducir la adivinación, el prompt incluye junto a cada canción los *tags específicos de la canción que Last.fm tiene curados por la comunidad* (endpoint `track.gettoptags`) — etiquetas como "spanish, paraguayo, cumbia" o "english, ballad, acoustic" que dan señales directas del idioma y el género. Si Last.fm no tiene tags para la canción (pistas oscuras), se usa como fallback los tags del artista (`artist.gettoptags`). Las canciones instrumentales o sin letra se clasifican como *"Sin letra"*.

____

No existe un catálogo cerrado de idiomas. El sistema reconoce cualquier idioma que ChatGPT pueda identificar. Esto permite generar playlists para idiomas que un catálogo fijo no contemplaría (ej: tailandés, suajili, catalán).

____

=== 3.2 Géneros Musicales

|===
|# |Género |Incluye 

|1 |Pop |(ej: Música comercial y popular de todo tipo)
|2 |Rock |(ej: Metal, punk, alternativo, grunge)
|3 |Hip-Hop / Rap |(ej: Rap, trap, lo-fi hip-hop)
|4 |Electrónica / Dance |(ej: EDM, house, techno, trance, dubstep)
|5 |R&amp;B / Soul / Funk |(ej: R&amp;B moderno, soul, funk)
|6 |Jazz / Blues |(ej: Jazz clásico, smooth jazz, blues, jazz fusion)
|7 |Clásica / Orquestal |(ej: Música clásica, sinfónica, barroca, contemporánea)
|8 |Reggae / Ska / World |(ej: Reggae, dub, ska y música global)
|9 |Country / Folk |(ej: Country, folk, bluegrass, música tradicional)
|10 |Latina / Urbano |(ej: Salsa, bachata, reggaetón, cumbia, merengue)
|===

____

La clasificación de género se realiza mediante consultas a la API de ChatGPT (GPT-4.1-mini). El prompt incluye los tags de género que Last.fm asigna al artista principal (`artist.gettoptags`) — etiquetas curadas por la comunidad como "rock, indie rock" o "reggaeton, latin" — de modo que ChatGPT *mapea* esos tags a uno de los 10 cubos en vez de *adivinar* el género desde el nombre del artista.

____

=== 3.3 Artistas

Los artistas se obtienen directamente de la API de Spotify. Una canción puede tener múltiples artistas (colaboraciones), y todos se registran. No existe un catálogo fijo de artistas; se descubren dinámicamente a partir de las canciones de las playlists del usuario.

'''

=== 3.4 Nota sobre la precisión de la clasificación

IMPORTANTE: la clasificación automática de *género* e *idioma* depende de ChatGPT y de los tags de Last.fm, y **no es siempre exacta**. Fuentes de error esperables:

* *ChatGPT es un LLM no determinista:* aun con `temperature: 0` y reglas estrictas en el prompt, ocasionalmente devuelve un idioma o subgénero que no corresponde. La red de seguridad (whitelist de géneros + diccionario de alias + reintentos) captura la mayoría de casos pero no todos.
* *Los tags de Last.fm son curados por la comunidad:* pueden tener sesgo, estar incompletos para canciones oscuras, o directamente no existir. Si no hay tags ni de canción ni de artista, ChatGPT depende solo del título y el nombre del artista.
* *Tags de artista en nombres ambiguos:* un artista con nombre corto o común (ej: "Rei") puede compartir identificador en Last.fm con otro artista más popular que tiene tags distintos. El fallback combina tags de todos los artistas de una colaboración para mitigar esto, pero no lo elimina.

El usuario puede corregir manualmente una clasificación errónea editando la playlist directamente en Spotify (ej: si Splitify mandó una canción a "Sin letra" por error, el usuario la agrega manualmente a la playlist de idioma correcta). Las ediciones manuales en Spotify se respetan en futuros refreshes (`manuallyAdded=true`).

== 5. Requerimientos Funcionales

=== RF1 — Autenticación con Spotify

*Descripción:* El sistema permitirá al usuario iniciar sesión mediante Spotify OAuth y cerrar sesión.

*Criterios de aceptación:*
- El usuario puede autenticarse con sus credenciales de Spotify.
- El usuario puede cerrar sesión en cualquier momento.
- La sesión se mantiene activa mientras el usuario navega la aplicación.
- Una vez autenticado, el header de la aplicación muestra la foto de perfil y el nombre del usuario obtenidos de la API de Spotify (`GET /me`).

'''

=== RF1.1 — Sincronización de Playlists

*Descripción:* Antes de poder operar con sus playlists, el usuario deberá sincronizarlas desde Spotify. La primera vez que el usuario accede al sistema (o cuando no existen playlists sincronizadas), se mostrará un estado vacío con un botón *"Sincronizar Playlists"*. Al presionarlo, el sistema mostrará una pantalla bloqueante de sincronización que impide cualquier otra interacción hasta que el proceso finalice.

*Flujo:*
1. El usuario inicia sesión y accede a la página principal.
2. Si no existen playlists sincronizadas, se muestra un estado vacío con el botón "Sincronizar Playlists". Los demás botones de acción (ej: "Organizar") se muestran deshabilitados.
3. El usuario presiona "Sincronizar Playlists".
4. Se muestra una pantalla bloqueante con el mensaje "Sincronizando..." que impide cualquier navegación o interacción.
5. El sistema obtiene todas las playlists del usuario desde Spotify (incluyendo Liked Songs) y, para cada una, obtiene *todas* sus canciones con metadatos completos (nombre, artista, álbum, fecha de lanzamiento). La descarga se realiza paginando la API de Spotify con `limit=50` y siguiendo el campo `next` de cada respuesta hasta agotar los resultados. Las canciones se clasifican con ChatGPT (género e idioma) durante la sincronización.
6. Al finalizar, se redirige al usuario a la página principal con las playlists ya cargadas.

*Criterios de aceptación:*
- Si no hay playlists sincronizadas, la página principal muestra un estado vacío con el botón "Sincronizar Playlists".
- Los botones de acción se muestran deshabilitados mientras no haya playlists sincronizadas.
- Durante la sincronización, se muestra una pantalla bloqueante que impide cualquier interacción.
- Al finalizar la sincronización, las playlists se muestran en la página principal.

'''

=== RF2 — Visualización y Organización de Playlists

*Descripción:* Una vez sincronizadas las playlists (RF1.1), el sistema mostrará todas las playlists del usuario en la página principal, organizadas en dos secciones:

* *Sección superior — Playlists del usuario:* playlists creadas por el usuario en Spotify. Cada playlist cuenta con una casilla de selección que permite marcarla como origen para la generación de nuevas playlists (RF4–RF7). Se incluye un botón *"Seleccionar todas"* para marcar o desmarcar todas las playlists de esta sección simultáneamente.
* *Sección inferior — Playlists de Splitify:* playlists generadas previamente por el sistema. Cada playlist dispone de:
** Un botón *"Eliminar"* para eliminarla de la cuenta de Spotify del usuario.
** Un botón *"Actualizar"* para sincronizar su contenido con las playlists origen actuales (ver RF2.2).
** Una casilla de selección para operaciones en lote.

El usuario puede acceder a cualquier playlist (de usuario o de Splitify) para consultar las canciones con su información detallada.

*Criterios de aceptación:*
- Se muestran todas las playlists del usuario en la sección superior de la página principal.
- La playlist de Liked Songs se incluye en la sección de playlists del usuario.
- Las playlists generadas por Splitify se muestran en la sección inferior, separadas visualmente de las del usuario.
- Cada playlist del usuario cuenta con una casilla de selección y existe un botón "Seleccionar todas".
- Al acceder a una playlist, se muestran las canciones con su información (nombre, artista, año, género).
- Cada playlist de Splitify dispone de los botones "Eliminar" y "Actualizar".

'''

=== RF2.1 — Gestión en Lote de Playlists de Splitify

*Descripción:* El sistema permitirá al usuario realizar operaciones en lote sobre las playlists generadas por Splitify. El usuario podrá seleccionar múltiples playlists de Splitify mediante casillas de selección individuales o mediante un botón *"Seleccionar todas"*. Una vez realizada la selección, dispondrá de dos acciones:

* *"Eliminar seleccionadas":* elimina de la cuenta de Spotify del usuario todas las playlists de Splitify seleccionadas.
* *"Actualizar seleccionadas":* sincroniza el contenido de todas las playlists de Splitify seleccionadas con las playlists origen actuales.

*Criterios de aceptación:*
- Cada playlist de Splitify cuenta con una casilla de selección individual.
- Existe un botón "Seleccionar todas" que marca o desmarca todas las playlists de Splitify.
- El botón "Eliminar seleccionadas" elimina únicamente las playlists de Splitify marcadas.
- El botón "Actualizar seleccionadas" sincroniza únicamente las playlists de Splitify marcadas.
- Los botones de acción en lote se habilitan solo cuando hay al menos una playlist seleccionada.

'''

=== RF2.2 — Actualización de Playlists de Splitify

*Descripción:* El sistema permitirá al usuario actualizar las playlists de Splitify para reflejar tanto los cambios ocurridos en las playlists origen como los cambios manuales realizados directamente en Spotify. La actualización puede realizarse de forma individual (botón "Actualizar" en cada playlist de Splitify) o en lote (botón "Actualizar seleccionadas" en RF2.1). Para esto, el sistema debe recordar qué playlists origen y qué criterio de organización (idioma, género, fecha de lanzamiento, o filtros personalizados) se usaron al crear cada playlist de Splitify.

*Flujo:*
1. El usuario presiona "Actualizar" en una playlist de Splitify (o selecciona varias y presiona "Actualizar seleccionadas").
2. El sistema genera una *vista previa* de los cambios detectados comparando el estado real en Spotify, la base de datos y las playlists origen.
3. El sistema detecta *dos tipos de divergencia* entre el hijo y sus fuentes, y por cada uno muestra un modal de conflictos con la misma estructura (checkboxes por canción, agrupación colapsable por playlist en lote, checkbox global "seleccionar todas", contador "X de Y"):
   a. *Canciones eliminadas manualmente desde Spotify que siguen en las playlists origen* (modo "restaurar"). Por defecto *todas marcadas* (lo más común es querer restaurar). El usuario confirma con "Restaurar seleccionadas (N)", descarta con "No restaurar ninguna", o cancela. Las *no seleccionadas* se marcan como excluidas permanentemente.
   b. *Canciones que siguen en el hijo pero cuyo origen las quitó* (porque el usuario las eliminó del origen, o porque la playlist origen se vació o se borró de Spotify) (modo "quitar"). Por defecto *ninguna marcada* (lo más común es querer conservarlas). El usuario confirma con "Quitar seleccionadas del hijo (N)", descarta con "No quitar ninguna", o cancela. Las canciones *seleccionadas* se eliminan del hijo; las *no seleccionadas* se conservan y pasan a tratarse como agregadas manualmente (no se vuelve a preguntar por ellas en refreshes futuros).
   Si ambos tipos de divergencia coexisten, los modales se muestran de forma secuencial (primero restaurar, luego quitar) y al final se aplica una única actualización con ambas decisiones.
4. El sistema construye la lista final de canciones aplicando las siguientes reglas:
   - *Canciones de las playlists origen:* se mantienen si siguen existiendo en las playlists origen y cumplen el criterio.
   - *Canciones agregadas manualmente por el usuario en Spotify:* se conservan permanentemente en actualizaciones sucesivas, independientemente de si existen en las playlists origen.
   - *Canciones eliminadas manualmente por el usuario desde Spotify:* el sistema pregunta (modo "restaurar"). Si el usuario elige no restaurar una canción específica, se marca como excluida y no se vuelve a agregar ni a preguntar. Si el usuario vuelve a agregar manualmente en Spotify una canción previamente excluida, la exclusión se revoca automáticamente.
   - *Canciones sin origen (playlist origen vaciada, eliminada de Spotify, o que quitó la canción):* el sistema pregunta (modo "quitar"). Por defecto se conservan. Las que el usuario decide conservar se convierten automáticamente en "manualmente agregadas" (el vínculo con la fuente desaparecida se rompe y no se vuelve a preguntar). Las que el usuario decide quitar se eliminan del hijo.
   - *Canciones nuevas en las playlists origen:* si son verdaderamente nuevas (nunca estuvieron en la playlist de Splitify ni fueron excluidas), se clasifican (género e idioma) y, si cumplen con el criterio de la playlist, se agregan.
   - *Playlist origen eliminada:* si una de las playlists origen ya no existe, se elimina la referencia a esa fuente (el vínculo en `SplitifyPlaylistSource`), pero el hijo NO se auto-elimina — sus canciones pasan por el flujo de "canciones sin origen" descrito arriba.
5. Se clasifica cualquier canción sin género/idioma (ej: canciones agregadas manualmente desde Spotify).
6. Se reordena la playlist completa por fecha de lanzamiento (RN1) y se reemplaza el contenido en Spotify para corregir el orden.
7. Adicionalmente, el sistema sincroniza el *nombre* y la *imagen* de la playlist de Splitify con el estado actual en Spotify (por si el usuario las editó directamente desde Spotify entre actualizaciones).
8. Si tras la actualización la playlist de Splitify queda sin canciones (típicamente porque el usuario marcó todas sus canciones sin origen para quitar), se elimina automáticamente tanto de Spotify como de la base de datos.
9. Si la playlist de Splitify fue eliminada desde Spotify (ya no existe en la cuenta del usuario), se detecta automáticamente y se elimina de la base de datos.

*Criterios de aceptación:*
- El sistema registra qué playlists origen y qué criterio de organización se usaron para crear cada playlist de Splitify (incluyendo los filtros seleccionados en modo personalizado).
- Antes de actualizar, el sistema muestra una vista previa con las dos categorías de divergencia (canciones eliminadas manualmente desde Spotify, y canciones cuyo origen desapareció), cada una con un modal propio de checkboxes individuales por canción.
- En actualizaciones en lote, las canciones del modal se agrupan por playlist con cabeceras colapsables para facilitar la diferenciación.
- El usuario puede elegir exactamente qué canciones restaurar y qué canciones quitar del hijo (granularidad por canción, no todo-o-nada, en ambos modos).
- Los defaults de los modales reflejan el comportamiento más común: modo "restaurar" viene con *todas marcadas*, modo "quitar" con *ninguna marcada*.
- Si el usuario elige no restaurar una canción específica, la decisión se recuerda permanentemente (campo `excluded` en la entidad Song). No se vuelve a preguntar en actualizaciones futuras.
- Si el usuario elige conservar una canción que perdió su origen, esa canción pasa a ser tratada como agregada manualmente (campo `manuallyAdded=true`) — no se vuelve a preguntar por ella aunque el origen nunca reaparezca.
- Las canciones agregadas manualmente desde Spotify se conservan permanentemente (campo `manuallyAdded` en la entidad Song), no solo en la primera actualización.
- Las canciones que perdieron su origen *no se eliminan automáticamente del hijo*: el sistema siempre pregunta antes (excepto si el usuario ya confirmó quitarlas en una actualización previa).
- Si el usuario re-agrega en Spotify una canción previamente excluida, la exclusión se revoca automáticamente.
- Las canciones nuevas de las playlists origen que cumplan el criterio se agregan automáticamente.
- Tras la actualización, todas las canciones se reordenan por fecha de lanzamiento en Spotify.
- Tras la actualización, el nombre e imagen de la playlist se sincronizan con Spotify si cambiaron.
- Si una playlist origen fue eliminada, se elimina la referencia a esa fuente (pero las canciones del hijo provenientes de ella entran al flujo de "canciones sin origen" en el próximo refresh).
- Si la playlist de Splitify queda vacía tras la actualización (típicamente porque el usuario marcó todas las canciones sin origen para quitar), se elimina automáticamente de Spotify y de la base de datos.
- Si la playlist de Splitify fue eliminada desde Spotify, se detecta y se elimina de la base de datos.
- La actualización se refleja tanto en Spotify como en la base de datos local.

'''

=== RF2.3 — Edición de Metadatos de Playlists de Splitify

*Descripción:* El sistema permitirá al usuario cambiar el nombre y la foto de portada de cualquier playlist de Splitify directamente desde la página principal. Estas acciones son *exclusivas* de playlists de Splitify (no se aplican a playlists originales del usuario, que son read-only desde la app). Los cambios se aplican inmediatamente tanto en la base de datos local como en Spotify, sin necesidad de re-sincronizar.

*Flujo (renombrar):*
1. En la card de la playlist de Splitify, el usuario ve el nombre con un ícono de lápiz visible a su lado.
2. Al presionar el ícono, el nombre se transforma en un input editable con botones de confirmar (✓) y cancelar (✗) dentro del mismo borde.
3. El usuario edita el nombre y presiona Enter o el botón de confirmar (Escape cancela).
4. El sistema envía el nuevo nombre a Spotify (`PUT /v1/playlists/{id}`) y actualiza la base de datos.
5. El cambio se refleja instantáneamente en la UI.

*Flujo (cambiar foto):*
1. El usuario posa el cursor sobre la portada de una playlist de Splitify — aparece un overlay con el texto "Cambiar foto" y un ícono de cámara.
2. Al hacer clic, se abre el selector de archivos del sistema operativo (filtrado a imágenes).
3. El navegador redimensiona la imagen a máximo 640px (lado mayor) y la comprime a JPEG calidad 0.85 para cumplir el límite de Spotify de 256KB.
4. El sistema sube la imagen a Spotify (`PUT /v1/playlists/{id}/images`, JPEG en base64) y, tras unos segundos, obtiene la URL de la imagen ya procesada (Spotify puede demorar unos segundos en propagarla).
5. La foto de la card se actualiza con la nueva imagen inmediatamente.

*Criterios de aceptación:*
- Las playlists originales del usuario (sección superior) *no* tienen opción de renombrar ni cambiar foto desde la app.
- El ícono de lápiz para renombrar está siempre visible al lado del nombre (no requiere hover para descubrirlo).
- El input de edición respeta las dimensiones del card (no lo deforma ni se desborda).
- Al cambiar el nombre, el cambio aparece en Spotify y en la app al instante.
- Al pasar el cursor sobre la foto de una playlist de Splitify, aparece un indicador visual de que es editable.
- Al cambiar la foto, el sistema acepta formatos JPEG/PNG/WebP del usuario y los convierte a JPEG comprimido antes de enviar.
- Si Spotify rechaza la imagen (tamaño, formato), se muestra un mensaje de error al usuario en la card.
- El cambio de foto requiere el scope adicional `ugc-image-upload` en la autorización OAuth.

'''

=== RF3 — Obtención de Metadatos de Canciones

*Descripción:* El sistema obtendrá la información de cada canción en una playlist: id, nombre, artista, álbum, año de lanzamiento, fecha de lanzamiento, género e idioma. El género e idioma se obtienen mediante ChatGPT API (GPT-4.1-nano). El año y fecha de lanzamiento se obtienen directamente de Spotify (fecha del álbum). En la vista de detalle de una playlist, las canciones se cargarán de forma progresiva conforme el usuario haga scroll (*carga paginada / infinite scroll*), ya que una playlist puede contener cientos o miles de canciones.

*Criterios de aceptación:*
- Se obtienen todos los campos definidos en la sección 4.1 para cada canción.
- El género se clasifica correctamente en una de las 10 categorías definidas en la sección 3.2.
- El idioma se detecta dinámicamente mediante ChatGPT API (ver sección 3.1).
- El año de lanzamiento se toma directamente de Spotify (fecha del álbum).
- Las clasificaciones de género e idioma se cachean para evitar consultas repetidas en re-sincronizaciones.
- En la vista de detalle, las canciones se cargan de forma paginada conforme el usuario hace scroll, sin cargar todas de una vez.
- Se muestra un indicador de carga mientras se obtienen más canciones.

'''

=== RF4 — Ordenar por Idioma

*Descripción:* El usuario podrá crear nuevas playlists filtrando canciones por idioma.

*Flujo:*
1. El usuario selecciona una o más playlists origen (incluyendo Liked Songs) desde la sección superior de la página principal.
2. El sistema genera una playlist por cada idioma detectado entre las canciones de las playlists seleccionadas. Los idiomas se determinan dinámicamente a partir de la clasificación de ChatGPT (ver sección 3.1), sin limitarse a un catálogo fijo.
3. *Umbral mínimo de 5 canciones por idioma:* los idiomas que no alcanzan las 5 canciones no generan una playlist propia. Sus canciones se agrupan en una única playlist `Splitify Other Languages Songs` (ver RN10). Motivación: una playlist con 2 o 3 canciones se percibe como testimonial y satura la sección de Splitify sin aportar valor práctico.
4. Las canciones dentro de cada playlist se ordenan por fecha de lanzamiento (ascendente).

*Nomenclatura de playlists creadas:*
- `Splitify {Idioma} Songs` para idiomas con ≥ 5 canciones (ej: `Splitify Español Songs`, `Splitify Inglés Songs`).
- `Splitify Other Languages Songs` para el bucket de idiomas con < 5 canciones. Solo se crea si existe al menos un idioma por debajo del umbral entre las canciones seleccionadas.

'''

=== RF5 — Ordenar por Género

*Descripción:* El usuario podrá crear nuevas playlists filtrando canciones por género musical.

*Flujo:*
1. El usuario selecciona una o más playlists origen (incluyendo Liked Songs) desde la sección superior de la página principal.

== 6. Reglas de Negocio

|===
|# |Regla 

|RN1 |Todas las playlists creadas por el sistema estarán ordenadas por fecha de lanzamiento de las canciones (ascendente).
|RN2 |Los idiomas se detectan dinámicamente mediante ChatGPT API. Se genera una playlist por cada idioma detectado sin limitarse a un catálogo fijo. Las canciones instrumentales o sin letra se agrupan en una playlist "Sin letra".
|RN3 |La clasificación de género e idioma se delega a ChatGPT API (GPT-4.1-mini). Para mejorar la precisión, antes de llamar a ChatGPT el sistema obtiene de *Last.fm* los tags curados por la comunidad y los incluye entre corchetes en el user prompt (ej: `"Tití Me Preguntó" de Bad Bunny [tags de Last.fm: reggaeton, latin, spanish, 2022, puerto rican]`). Flujo de prefetch en dos fases:

* *Fase 1 — Track tags (preferido):* `track.gettoptags` por canción específica — devuelve tags ricos y precisos, con señales directas de idioma (`spanish`, `paraguayo`, `french`), mood (`ballad`, `acoustic`), época y género fino. Cacheado en `TrackTags` por `spotifyTrackId`.
* *Fase 2 — Artist tags (fallback):* si Last.fm no tiene tags para la canción específica (pistas oscuras), se usan los tags del artista como señal más débil. Cacheados en `ArtistGenres` por nombre de artista.

Esos tags están basados en datos reales de la comunidad, por lo que ChatGPT pasa de *adivinar* el género a *mapear* esos tags a uno de los 10 cubos — el prompt incluye mapeos explícitos para tags comunes (Soundtrack/Score/Orchestral → Clásica/Orquestal, K-pop/J-pop → Reggae/Ska/World, Reggaeton/Cumbia/Vallenato → Latina/Urbano, etc.). Los tags también sirven como pista para el idioma: `"spanish"` → Español, `"french"` → Francés, `"paraguayo"` → Español. Ambas cachés persisten entre syncs.

*Por qué Last.fm y no Spotify:* hasta febrero 2026 usábamos el endpoint `/v1/artists?ids=...` de Spotify. Ese endpoint fue *removido* en feb 2026, y además Spotify eliminó el campo `genres` del objeto Artist. Last.fm mantiene una base de tags equivalente, gratis y estable. Requiere registrar una API key (env var `LASTFM_API_KEY`).

*Fallback pragmático:* ChatGPT a veces devuelve subgéneros literales fuera del catálogo (ej: "Vallenato", "Indie/Alternativo", "Latin/Urbano"). Antes de descartar la respuesta, el backend intenta mapearla contra un diccionario de *alias* (~90 entradas cubriendo typos, acentos omitidos y subgéneros comunes). Solo si el alias tampoco resuelve, la respuesta se descarta y la canción se reintenta. Lo mismo ocurre si ChatGPT devuelve `"languages": []` vacío — se fuerza el reintento para evitar guardar un idioma en blanco que contamine la UI. Las clasificaciones cacheadas parciales o con géneros inválidos se detectan al vuelo y se invalidan automáticamente. El año de lanzamiento se obtiene directamente de Spotify (fecha del álbum).

Durante la sincronización el sistema insiste hasta clasificar: barrido por lotes de 20 canciones con 2 reintentos internos ante excepción (timeout, rate limit, JSON roto), y un bucle externo de hasta 5 pasadas que re-envía a ChatGPT solo las canciones que siguen sin género (por excepción o respuesta incompleta sin excepción) — hasta 10 intentos reales por canción antes de rendirse. Si OpenAI queda caída toda la sync, las canciones irresueltas quedan con género/idioma nulos en BD; la sincronización termina sin error y esas canciones aparecen como `-` en la vista de detalle.
|RN9 |Ninguna acción de creación o filtrado puede ejecutarse con canciones sin clasificar. Antes de organizar (RF4/RF5/RF6/RF6.1), crear playlist personalizada (RF7) o calcular los filtros disponibles, el sistema verifica que todas las canciones involucradas tengan género e idioma. Si hay huecos, hace un intento corto adicional de clasificación (2 pasadas) y, si aún quedan sin clasificar, devuelve error HTTP 422 con mensaje *"Hay N canciones sin clasificar. ChatGPT no respondió a tiempo, intentá de nuevo en unos segundos."* El frontend muestra ese mensaje en un modal de error; el usuario puede reintentar la acción sin necesidad de re-sincronizar. Esto garantiza que nunca se creen playlists con canciones faltantes silenciosamente.
|RN6 |Los filtros del modo personalizado (RF7) son dinámicos: solo se muestran los idiomas, géneros y artistas que existen en las canciones de las playlists seleccionadas. No se presentan opciones que no producirían resultados.
|RN7 |El filtrado de canciones es *inclusivo* en campos multi-valor: si una canción tiene múltiples idiomas (ej: inglés y español) y el usuario filtra por inglés, la canción se incluye. De igual forma, si una canción es una colaboración de varios artistas y el usuario filtra por uno de ellos, basta con que el artista deseado participe en la canción para que se incluya en el resultado.
|RN4 |La creación de playlists genéricas (RF4, RF5, RF6) siempre producirá al menos una playlist, dado que toda canción pertenece a un idioma, un género y una fecha de lanzamiento.
|RN5 |La creación de playlists personalizadas (RF7) puede no generar ninguna playlist si ninguna canción cumple con todos los filtros seleccionados. En ese caso, el sistema notificará al usuario mediante un mensaje de error indicando que no se encontraron canciones que coincidan con los criterios especificados.
|RN8 |El sistema debe persistir la relación entre cada playlist de Splitify y sus playlists origen, así como el criterio de organización utilizado (idioma, género, fecha de lanzamiento, o filtros personalizados de RF7). Para playlists personalizadas (RF7) se persisten las listas completas de idiomas, géneros y artistas elegidos originalmente. Esta información es necesaria para la funcionalidad de actualización (RF2.2): permite al sistema re-evaluar las canciones de las fuentes y aplicar el mismo criterio original.
|RN10 |Al ordenar por idioma (RF4 y RF6.1), un idioma debe tener al menos *5 canciones* entre las canciones seleccionadas para obtener una playlist dedicada. Los idiomas que no alcanzan el umbral se consolidan en una única playlist `Splitify Other Languages Songs`. El bucket usa un tipo de filtro propio (`otherLanguages`) que persiste la lista de idiomas agrupados, de modo que al actualizar (RF2.2) el sistema sigue incluyendo únicamente canciones que pertenecen al conjunto original de idiomas minoritarios. El bucket se crea incluso si solo un idioma quedó por debajo del umbral.
