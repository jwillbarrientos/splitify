== 2. Visión General de la Arquitectura

Splitify sigue una arquitectura *cliente-servidor* con separación clara entre frontend y backend:

* *Frontend:* Single Page Application (SPA) desarrollada con React, JavaScript y Vite. Se comunica con el backend exclusivamente mediante una API REST.
* *Backend:* API REST desarrollada con Spring Boot (Java 25). Gestiona la lógica de negocio, la autenticación, la persistencia y la comunicación con APIs externas.
* *APIs externas:* Spotify Web API (autenticación, lectura/escritura de playlists, fechas de lanzamiento) y ChatGPT API (clasificación de género e idioma).

== 5. Arquitectura del Backend

=== 5.1 Capas

El backend sigue una arquitectura en capas:

* *Controller:* recibe las peticiones HTTP, valida la entrada y delega al servicio correspondiente. Todos los endpoints se exponen bajo `/api/**`.
* *Service:* contiene la lógica de negocio. Coordina la obtención de datos de Spotify, la clasificación con ChatGPT y la creación de playlists.
* *Repository:* acceso a datos mediante Spring Data JPA.

=== 5.2 Servicios Principales

|===
|Servicio |Responsabilidad

|`SpotifyService` |Servicio principal: comunicación con la Spotify Web API (sincronización, CRUD de playlists, obtención de canciones), lógica de negocio (filtrado por idioma/género/fecha, creación y actualización de playlists de Splitify), y persistencia. Gestiona la actualización inteligente de playlists: detecta cambios manuales en Spotify, mantiene estado de exclusiones y adds manuales, y reordena por fecha de lanzamiento.
|`ClassificationService` |Comunicación con la API de OpenAI (GPT-4.1-nano) para clasificar el género e idioma de cada canción. El idioma se detecta dinámicamente sin catálogo fijo. Utiliza caché global (`SongClassification`) para evitar re-clasificar canciones ya procesadas.
|===

== 6. Arquitectura del Frontend

=== 6.1 SPA y Navegación

El frontend es una Single Page Application con navegación del lado del cliente gestionada por *React Router v7*. Spring Boot sirve el `index.html` generado por Vite para cualquier ruta que no comience con `/api/`. La navegación entre páginas no recarga el navegador. Las rutas autenticadas (`/` y `/playlist/:id`) comparten un layout (`AuthLayout` en `App.jsx`) que contiene el componente `Header` reutilizable y gestiona la obtención del perfil del usuario y el cierre de sesión.

=== 6.2 Páginas Principales

|===
|Página |Ruta |Descripción

|Login |`/login` |Botón de inicio de sesión con Spotify
|Principal |`/` |Estado vacío con botón "Sincronizar Playlists" si no hay playlists sincronizadas. Una vez sincronizadas: sección superior (playlists del usuario con selección) y sección inferior (playlists de Splitify con acciones individuales y en lote)
|Sincronizando |`/` (modal bloqueante) |Pantalla bloqueante que se muestra durante la sincronización de playlists, impidiendo cualquier interacción
|Detalle de playlist |`/playlist/:id` |Lista de canciones con metadatos (nombre, artista, año, género, idioma) con carga paginada por scroll (infinite scroll)
|Crear playlist personalizada |`/create` |Formulario con filtros dinámicos de idioma, género y artista. Solo muestra los filtros correspondientes a idiomas, géneros y artistas presentes en las canciones de las playlists seleccionadas (RF7)
|===

=== 6.3 Estilos

La interfaz utiliza *Tailwind CSS* como framework de utilidades. No se emplea una librería de componentes pre-construidos; todos los componentes se construyen manualmente con clases de utilidad de Tailwind, lo que permite un control total sobre el diseño.

=== 6.4 Comunicación con el Backend

Todas las llamadas al backend se realizan mediante `fetch`, apuntando a `/api/**`. En desarrollo, Vite redirige estas peticiones al servidor de Spring Boot mediante su proxy integrado.

==== 7.1.1 Restricciones del Modo Desarrollo de Spotify

Desde febrero 2026, Spotify impone restricciones en apps con *Development Mode*:

* El endpoint `GET /playlists/{id}/items` solo devuelve canciones de playlists que el usuario **creó o colabora**. Playlists de otros usuarios (que solo se siguen) devuelven 403 Forbidden.
* El endpoint `POST /users/{id}/playlists` fue reemplazado por `POST /me/playlists`.
* El endpoint `POST /playlists/{id}/tracks` fue renombrado a `POST /playlists/{id}/items`.
* El campo `track` dentro de cada item fue renombrado a `item` en la respuesta JSON.
* El campo `tracks` en el objeto playlist fue renombrado a `items`.
* Máximo 5 usuarios autorizados (deben agregarse manualmente en el Developer Dashboard).
* Se requiere cuenta Premium para el propietario de la app.

Para acceso completo (incluyendo playlists de otros usuarios), se debe solicitar *Extended Quota Mode* en el Spotify Developer Dashboard (requiere aprobación manual de Spotify).

=== 7.2 API de Clasificación (OpenAI)

El servicio `ClassificationService` se comunica con la API REST de OpenAI mediante `RestClient` de Spring. El modelo utilizado es *GPT-4.1-nano*, seleccionado por ser suficiente para tareas de categorización y por su bajo costo. ChatGPT se utiliza *exclusivamente* para clasificar género e idioma (no para fechas de lanzamiento).

El servicio envía el nombre y artista de cada canción para obtener:

* *Género:* clasificado en una de las 10 categorías definidas en el SRS (sección 3.2).
* *Idioma(s):* detectados dinámicamente sin limitarse a un catálogo fijo. ChatGPT identifica el/los idioma(s) de la canción. Una canción puede tener múltiples idiomas si se canta en más de uno (ej: una canción con partes en inglés y español devuelve ambos idiomas). Las canciones instrumentales se clasifican como "Sin letra".

Para optimizar costos y rendimiento, las clasificaciones se almacenan en caché en la base de datos. Si una canción ya fue clasificada previamente, no se consulta la API nuevamente.

└── language: String (nullable)           ← Idioma(s) clasificados

├── id: Long (PK, autogenerado)

└── sourcePlaylistId: Long (FK → Playlist)    ← Playlist origen del usuario

La tabla `SplitifyPlaylistSource` permite al sistema saber qué playlists origen se usaron para crear cada playlist de Splitify, lo cual es necesario para la funcionalidad de actualización (RF2.2). Junto con los campos `filterType` y `filterValue` de la entidad `Playlist`, el sistema puede re-evaluar las canciones de las fuentes y aplicar el mismo criterio de organización original.

=== 8.2 Estrategia de Persistencia

* La entidad `Playlist` persiste las playlists del usuario de Spotify localmente tras la sincronización. Al re-sincronizar, se eliminan los datos viejos y se reemplazan con datos frescos de Spotify.
* La entidad `Song` almacena todas las canciones de cada playlist. Una misma canción de Spotify puede existir múltiples veces si aparece en varias playlists (relación ManyToOne simple). Los campos `releaseYear` y `releaseDate` se obtienen directamente de Spotify (fecha del álbum). Los campos `genre` y `language` se clasifican mediante ChatGPT API (GPT-4.1-nano). En playlists de Splitify, los campos `excluded` y `manuallyAdded` rastrean el estado de cada canción: `excluded=true` indica que el usuario eligió no restaurar la canción (se recuerda permanentemente); `manuallyAdded=true` indica que el usuario la agregó directamente en Spotify (se preserva aunque no esté en las fuentes). Canciones con `excluded=true` no se muestran en la vista de detalle ni se agregan a Spotify.
+
NOTE: Durante la fase de desarrollo, la sincronización se limita a las primeras 10 canciones por playlist para agilizar las pruebas. En la versión final se traerán todas las canciones (paginando con `limit=50` y siguiendo el campo `next` de la respuesta de Spotify).
* La entidad `SongClassification` funciona como caché global de clasificaciones. Almacena el género e idioma clasificados por ChatGPT indexados por `spotifyId`. Al re-sincronizar, las clasificaciones existentes se preservan y solo se consulta ChatGPT para canciones nuevas.
* La entidad `SplitifyPlaylistSource` registra la relación entre cada playlist de Splitify y sus playlists origen. Junto con los campos `filterType` y `filterValue` de `Playlist`, esta tabla permite al sistema re-evaluar las canciones de las fuentes y aplicar el mismo criterio original durante la actualización (RF2.2).
* Liked Songs se almacena como una playlist con `spotifyId = "liked_songs"` e `imageUrl = null`. La consulta de playlists ordena siempre Liked Songs primero mediante `ORDER BY CASE WHEN spotifyId = 'liked_songs' THEN 0 ELSE 1 END, id ASC`.

=== 9.3 CSRF y CORS

* *CORS:* En desarrollo, el proxy de Vite redirige las peticiones `/api/**` al backend, por lo que el navegador las ve como mismo origen y no se genera conflicto de CORS. En producción, frontend y backend se sirven desde el mismo JAR (mismo origen), por lo que CORS no aplica. No se requiere configuración adicional de CORS.
* *CSRF:* La protección CSRF está deshabilitada para las rutas `/api/**`. Dado que la SPA y la API se sirven desde el mismo origen (mismo JAR en producción, proxy en desarrollo) y la sesión usa cookies con `SameSite=Lax`, el riesgo de CSRF es mínimo. Esto simplifica la comunicación frontend-backend al no requerir envío de tokens CSRF.

== 10. Integración Frontend-Backend (Build y Desarrollo)

=== 10.1 Desarrollo Local

En desarrollo se ejecutan dos procesos en paralelo:

* `./gradlew bootRun` — Backend en el puerto *8080*
* `npm run dev` (dentro de `frontend/`) — Vite dev server en el puerto *5173* con Hot Module Replacement

Vite está configurado con un proxy que redirige las peticiones `/api/**` al puerto 8080, eliminando problemas de CORS durante el desarrollo (ver sección 9.3).

=== 10.2 Build de Producción

Gradle orquesta el build completo mediante el plugin `com.github.node-gradle.node`:

[source]
----
./gradlew build
    │
    ├── 1. npm install (frontend/)
    ├── 2. npm run build (frontend/)
    │       └── Vite genera archivos en src/main/resources/static/
    ├── 3. Compilación Java
    └── 4. Empaquetado JAR
            └── JAR único contiene backend + frontend estático

El resultado es un *único archivo JAR* que sirve tanto la API como el frontend estático.

=== 10.3 Configuración de Vite

El archivo `frontend/vite.config.js` define:

* *`build.outDir`*: `../src/main/resources/static` — deposita el build donde Spring Boot sirve archivos estáticos.
* *`server.proxy`*: redirige `/api` al backend durante desarrollo.

=== 10.4 SPA Forwarding

Un controller en Spring Boot reenvía todas las rutas no-API al `index.html`, permitiendo que React maneje la navegación del lado del cliente. Se utilizan dos mappings: uno para rutas de un segmento (`/login`) y otro para rutas anidadas (`/playlist/123`), excluyendo rutas de assets, API y OAuth:

[source,java]
----
@Controller
public class SpaController {
    @GetMapping(value = "/{path:[^\\.]*}")
    public String forward() {
        return "forward:/index.html";
    }

    @GetMapping(value = "/{path:(?!api|assets|oauth2|login\\/oauth2)[^\\.]*}/**")
    public String forwardNested() {
        return "forward:/index.html";
    }
}
----
