== 2. Visión General de la Arquitectura

Splitify sigue una arquitectura *cliente-servidor* con separación clara entre frontend y backend:

* *Frontend:* Single Page Application (SPA) desarrollada con React, JavaScript y Vite. Se comunica con el backend exclusivamente mediante una API REST.
* *Backend:* API REST desarrollada con Spring Boot (Java 25). Gestiona la lógica de negocio, la autenticación, la persistencia y la comunicación con APIs externas.
* *APIs externas:* Spotify Web API (autenticación, lectura/escritura de playlists) y ChatGPT API (clasificación de género musical).

== 5. Arquitectura del Backend

=== 5.1 Capas

El backend sigue una arquitectura en capas:

* *Controller:* recibe las peticiones HTTP, valida la entrada y delega al servicio correspondiente. Todos los endpoints se exponen bajo `/api/**`.
* *Service:* contiene la lógica de negocio. Coordina la obtención de datos de Spotify, la clasificación con ChatGPT y la creación de playlists.
* *Repository:* acceso a datos mediante Spring Data JPA.

=== 5.2 Servicios Principales

|===
|Servicio |Responsabilidad

|`SpotifyService` |Comunicación con la Spotify Web API: obtener playlists del usuario, obtener canciones de una playlist, crear playlists, agregar canciones, eliminar playlists.
|`ClassificationService` |Comunicación con la API de ChatGPT para obtener el año original de lanzamiento, género e idioma de cada canción a partir de su nombre y artista. El año se solicita a ChatGPT (en lugar de usar el de Spotify) porque Spotify puede devolver la fecha de una reedición o remasterización. El idioma se detecta dinámicamente (sin catálogo fijo). Gestiona el caché de clasificaciones para evitar consultas repetidas.
|`PlaylistService` |Lógica de negocio central: filtrado por idioma/género/época, aplicación de reglas de negocio (RN1–RN5), orquestación de la creación y actualización de playlists.
|===

=== 5.3 Endpoints de la API REST

|===
|Método |Ruta |Descripción |RF

|GET |`/api/auth/status` |Estado de la sesión del usuario |RF1
|POST |`/api/auth/logout` |Cerrar sesión |RF1
|POST |`/api/playlists/sync` |Sincronizar playlists del usuario desde Spotify (operación bloqueante) |RF1.1
|GET |`/api/playlists` |Obtener todas las playlists del usuario (Spotify + Splitify) |RF2
|GET |`/api/playlists/{id}/songs?page={n}&size={s}` |Obtener canciones de una playlist con metadatos (paginado) |RF2, RF3
|POST |`/api/playlists/create/by-language` |Crear playlists ordenadas por idioma |RF4
|POST |`/api/playlists/create/by-genre` |Crear playlists ordenadas por género |RF5
|POST |`/api/playlists/create/by-release-date` |Crear playlist ordenada por fecha de lanzamiento |RF6
|POST |`/api/playlists/create/combined` |Crear playlists de múltiples tipos genéricos a la vez |RF6.1
|GET |`/api/playlists/available-filters?playlistIds={ids}` |Obtener idiomas, géneros y décadas disponibles en las playlists seleccionadas |RF7
|POST |`/api/playlists/create/custom` |Crear playlist personalizada con filtros combinados |RF7
|PUT |`/api/playlists/splitify/{id}/refresh` |Actualizar una playlist de Splitify |RF2
|PUT |`/api/playlists/splitify/batch/refresh` |Actualizar múltiples playlists de Splitify |RF2.1
|DELETE |`/api/playlists/splitify/{id}` |Eliminar una playlist de Splitify |RF2
|DELETE |`/api/playlists/splitify/batch` |Eliminar múltiples playlists de Splitify |RF2.1
|===

'''

== 6. Arquitectura del Frontend

=== 6.1 SPA y Navegación

El frontend es una Single Page Application con navegación del lado del cliente gestionada por *React Router v7*. Spring Boot sirve el `index.html` generado por Vite para cualquier ruta que no comience con `/api/`. La navegación entre páginas no recarga el navegador.

=== 6.2 Páginas Principales

|===
|Página |Ruta |Descripción

|Login |`/login` |Botón de inicio de sesión con Spotify
|Principal |`/` |Estado vacío con botón "Sincronizar Playlists" si no hay playlists sincronizadas. Una vez sincronizadas: sección superior (playlists del usuario con selección) y sección inferior (playlists de Splitify con acciones individuales y en lote)
|Sincronizando |`/` (modal bloqueante) |Pantalla bloqueante que se muestra durante la sincronización de playlists, impidiendo cualquier interacción
|Detalle de playlist |`/playlist/:id` |Lista de canciones con metadatos (nombre, artista, año, género, idioma) con carga paginada por scroll (infinite scroll)
|Crear playlist personalizada |`/create` |Formulario con filtros dinámicos de idioma, género y época. Solo muestra los filtros correspondientes a idiomas, géneros y décadas presentes en las canciones de las playlists seleccionadas (RF7)
|===

=== 6.3 Estilos

La interfaz utiliza *Tailwind CSS* como framework de utilidades. No se emplea una librería de componentes pre-construidos; todos los componentes se construyen manualmente con clases de utilidad de Tailwind, lo que permite un control total sobre el diseño.

=== 6.4 Comunicación con el Backend

Todas las llamadas al backend se realizan mediante `fetch`, apuntando a `/api/**`. En desarrollo, Vite redirige estas peticiones al servidor de Spring Boot mediante su proxy integrado.

'''

== 7. Integraciones Externas

=== 7.1 Spotify Web API

|===
|Operación |Endpoint de Spotify |Uso en el sistema

|Autenticación |OAuth 2.0 Authorization Code Flow |Inicio de sesión del usuario (RF1)
|Obtener playlists |`GET /me/playlists` |Listar playlists del usuario (RF2)
|Obtener Liked Songs |`GET /me/tracks` |Incluir Liked Songs como playlist seleccionable (RF2)
|Obtener canciones |`GET /playlists/{id}/tracks` |Leer canciones para clasificación y filtrado (RF3)
|Crear playlist |`POST /users/{id}/playlists` |Generar nuevas playlists (RF4–RF7)
|Agregar canciones |`POST /playlists/{id}/tracks` |Poblar las playlists creadas (RF4–RF7)
|Eliminar playlist |`DELETE /playlists/{id}/followers` |Eliminar playlists de Splitify (RF2, RF2.1)
|===

=== 7.2 API de Clasificación (OpenAI)

El servicio `ClassificationService` utiliza el *OpenAI Java SDK* (`com.openai:openai-java`) para comunicarse con la API de OpenAI. El modelo utilizado es *GPT-4.1-nano*, seleccionado por ser suficiente para tareas de categorización simple y por su bajo costo.

El servicio envía el nombre y artista de cada canción para obtener:

* *Año original de lanzamiento:* se solicita a ChatGPT en lugar de usar la fecha de Spotify, ya que Spotify puede devolver la fecha de una reedición o remasterización en vez de la del lanzamiento original.
* *Género:* clasificado en una de las 10 categorías definidas en el SRS (sección 3.2).
* *Idioma:* detectado dinámicamente sin limitarse a un catálogo fijo. ChatGPT identifica el idioma exacto de la canción (ej: español, coreano, tailandés, catalán). Las canciones instrumentales se clasifican como "Sin letra".

Para optimizar costos y rendimiento, las clasificaciones se almacenan en caché en la base de datos. Si una canción ya fue clasificada previamente, no se consulta la API nuevamente.

'''

=== 8.1 Entidades

[source]
----
Song
├── spotifyId: String (PK)       ← ID único de Spotify
├── name: String
├── artist: String
├── releaseYear: Integer
├── genre: String                ← Género clasificado (de los 10 definidos)

SplitifyPlaylist
├── id: Long (PK, autogenerado)
├── spotifyPlaylistId: String    ← ID de la playlist en Spotify
├── name: String
├── type: Enum                   ← LANGUAGE, GENRE, RELEASE_DATE, CUSTOM
├── userId: String               ← ID del usuario de Spotify propietario

├── createdAt: Timestamp
└── updatedAt: Timestamp
----

=== 8.2 Estrategia de Persistencia

* La entidad `Song` funciona como caché de clasificaciones. Evita consultas repetidas a la API de ChatGPT para canciones ya clasificadas.
* La entidad `SplitifyPlaylist` registra las playlists creadas por el sistema, permitiendo identificarlas en la sección inferior de la página principal y rastrear sus playlists origen para la funcionalidad de actualización.
* Las playlists del usuario de Spotify no se persisten localmente; se obtienen en tiempo real desde la Spotify API.

== 9. Autenticación y Seguridad

=== 9.1 Flujo de Autenticación

[source]
----
Usuario ──▶ [Botón "Iniciar sesión"] ──▶ Spotify OAuth
                                              │
                                    (autoriza permisos)
                                              │
Spotify ──▶ [Redirect con código] ──▶ Backend (Spring Security)
                                              │
                                    (intercambia código por token)
                                              │
Backend ──▶ [Sesión autenticada] ──▶ Frontend (acceso a /api/**)
----

=== 9.2 Reglas de Seguridad

|===
|Ruta |Acceso

|`/api/**` |Solo usuarios autenticados
|`/api/auth/status` |Público (para verificar estado de sesión)
|`/login`, `/oauth2/**` |Público (flujo OAuth)
|Cualquier otra ruta |Público (sirve el SPA)
|===

=== 9.3 CSRF y CORS

* *CORS:* En desarrollo, el proxy de Vite redirige las peticiones `/api/**` al backend, por lo que el navegador las ve como mismo origen y no se genera conflicto de CORS. En producción, frontend y backend se sirven desde el mismo JAR (mismo origen), por lo que CORS no aplica. No se requiere configuración adicional de CORS.
* *CSRF:* Spring Security habilita protección CSRF por defecto. Para SPAs con autenticación basada en sesión, se configura Spring para exponer el token CSRF como cookie (`XSRF-TOKEN`). El frontend lee esta cookie y la envía como cabecera (`X-XSRF-TOKEN`) en cada petición que modifique estado (POST, PUT, DELETE).

=== 9.4 Permisos de Spotify (Scopes)

|===
|Scope |Necesario para

|`playlist-read-private` |Leer playlists privadas del usuario
|`playlist-read-collaborative` |Leer playlists colaborativas
|`playlist-modify-public` |Crear y modificar playlists públicas
|`playlist-modify-private` |Crear y modificar playlists privadas
|`user-library-read` |Acceder a Liked Songs
|===

== 10. Integración Frontend-Backend (Build y Desarrollo)

=== 10.1 Desarrollo Local

En desarrollo se ejecutan dos procesos en paralelo:

* `./gradlew bootRun` — Backend en el puerto *8080*
* `npm run dev` (dentro de `frontend/`) — Vite dev server en el puerto *5173* con Hot Module Replacement

Vite está configurado con un proxy que redirige las peticiones `/api/**` al puerto 8080, eliminando problemas de CORS durante el desarrollo (ver sección 9.3).

=== 10.2 Build de Producción

Gradle orquesta el build completo mediante el plugin `com.github.node-gradle.node`:

[source]
----
./gradlew build
    │
    ├── 1. npm install (frontend/)
    ├── 2. npm run build (frontend/)
    │       └── Vite genera archivos en src/main/resources/static/
    ├── 3. Compilación Java
    └── 4. Empaquetado JAR
            └── JAR único contiene backend + frontend estático
----

El resultado es un *único archivo JAR* que sirve tanto la API como el frontend estático.

=== 10.3 Configuración de Vite

El archivo `frontend/vite.config.js` define:

* *`build.outDir`*: `../src/main/resources/static` — deposita el build donde Spring Boot sirve archivos estáticos.
* *`server.proxy`*: redirige `/api` al backend durante desarrollo.

=== 10.4 SPA Forwarding

Un controller en Spring Boot reenvía todas las rutas no-API al `index.html`, permitiendo que React maneje la navegación del lado del cliente:

[source,java]
----
@Controller
public class SpaController {
    @GetMapping(value = "/{path:[^\\.]*}")
    public String forward() {
        return "forward:/index.html";
    }
}
----
